# Documentation Index

This documentation set is intentionally split by audience.

## If You Just Want To Use Tehuti

1. `GETTING_STARTED.md` for setup and first run.
2. `USER_GUIDE.md` for commands and practical workflows.
3. `TROUBLESHOOTING.md` when something breaks.

## If You Operate Tehuti In Real Workflows

1. `OPERATOR_RUNBOOK.md` for monitoring, noise control, and stuck-task triage.
2. `CONFIGURATION.md` for policy and runtime behavior.
3. `API_CONTRACTS.md` for wire/web/tool/metrics envelopes.
4. `PARITY_MATRIX.md` for cross-surface behavior parity tracking.
5. `OUTPUT_POLICY_AUDIT_2026-02-12.md` for output-integrity expectations.

## If You Develop Tehuti

1. `CONFIGURATION.md` for runtime config contract.
2. `TOOLS.md` for tool system overview.
3. `API_CONTRACTS.md` for response/telemetry contracts.
4. `PARITY_MATRIX.md` for cross-surface parity status.
5. `EXECUTION_TRACKER.md` for current implementation status by stream.
6. `MAINTAINER_MAP.md` for feature-to-file ownership and safe change workflow.
7. `LOOP_STATE_TABLE.md` for explicit runtime transition policy.
8. `RETRY_BACKOFF_POLICY.md` for retry/backoff parameterization details.
9. `adr/0001-runtime-foundation-and-capability-parity.md` for foundational design decisions.
10. `RELEASE_NOTES_2026-02-12.md`, `SESSION_CHANGES_2026-02-12.md`, `RELEASE_NOTES_2026-02-13.md`, and `SESSION_CHANGES_2026-02-13.md` for historical context.
11. `SLO_BASELINES.md` for operational guardrails and alert thresholds.

Archived planning references are in `docs/archive/` and are not canonical status sources.

## Style Notes

These docs are written for humans first:
- direct language
- practical examples
- clear tradeoffs
- minimal jargon unless necessary
